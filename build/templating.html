Templating

Now that we know how to return a template response. Let's look into how templating works in Bridge.

Expressions

Using Dart expressions in the templates work just like string interpolation. Any script can easily be executed like so:

<h1>What is 1 + 1?</h1>
<p>Easy! It's ${1 + 1}</p>
<h1>What is 1 + 1?</h1>
<p>Easy! It's 2</p>

Everything that has been imported or exported publically in your dart script can be used in template expressions.

// lib/app/my_class.dart
class MyClass {
  String property = 'value';
}
<p>${new MyClass().property}</p>
<p>value</p>

Note: Complex expressions are parsed on the server, so your client entities will not be accessible using this method

Escape dollar signs with a back-slash:

<p>\$WAG</p>
<p>$WAG</p>

Passing local variables

Passing locals to the template is super easy:

return template('index', withData: {
  'key': 'value'
});
<p>$key</p>
<p>value</p>

Directives

Trivial logic can be performed with directives, like so:

<p>
  @if (true)
    This content will always be shown
  @else if (false)
    This content will never be shown
  @else
    Neither will this
  @end if

  @for (item in someList)
    $item
  @end for
</p>

Comments

Besides supporting HTML comments, Bridge templates have normal double-slash syntax for comments, that disappear when the template is parsed.

// Super important!
<p>Paragraph</p>
<p>Paragraph</p>

Pre Processors

If a template file has the extension .html, it will be parsed as a Bridge template. However, there is support for a few other formats. All the following will have equal output:

return template('index', withData: {
  'key': 'value'
});
// lib/templates/index.html
<p>$key</p>
# lib/templates/index.md
$key
// lib/templates/index.jade
p= key
{{-- lib/templates/index.hbs --}}
<p>{{ key }}</p>
<p>value</p>
